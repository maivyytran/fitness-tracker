**Project Write-Up: Converting a Streamlit App to a Live Django Application**

For this phase of the project, I set out to convert my existing Streamlit fitness tracker into a professional Django web application and deploy it to a live URL.  I set out to build a full-stack web application that a fitness studio could use to manage its day-to-day operations. The goal was to create a system where staff could track members, instructors, classes, membership plans, and class registrations all in one place. I started with a Streamlit app connected to a PostgreSQL database, then converted the entire project into a Django application and deployed it to the web.

The conversion process started smoothly. I scaffolded the Django project, connected it to my existing PostgreSQL database using `managed = False` on all five models so Django would recognize the tables without recreating them, and registered everything in the admin panel. Seeing my existing data show up in the Django admin on the first try was a good sign. Building the six frontend pages went well once I understood how Django views, URLs, and templates work together compared to Streamlit's single-file approach.

Deployment is where everything got difficult. My first attempt was PythonAnywhere, which seemed straightforward until I discovered that the free tier blocks outgoing connections to external databases. Since my PostgreSQL database is hosted on Retool's servers, PythonAnywhere could not connect to it at all, which meant every login attempt returned a 500 error. I had to abandon PythonAnywhere entirely and switch to Railway mid-deployment, which cost significant time reconfiguring environment variables, the Procfile, and the WSGI setup.

Railway introduced its own challenges. The platform kept automatically running `createsuperuser` commands on every deploy, causing repeated failures. I had to add a `nixpacks.toml` file and manually set the start command to stop Railway from running its own build steps. There was also a persistent issue where Django's template caching on Railway kept serving an old version of `classes.html` that contained an invalid `split` filter. No matter how many times I edited and pushed the file, Railway continued serving the broken cached version. The only fix that worked was deleting the file from git entirely, pushing the deletion, then recreating and pushing the file fresh.

If I were to do this again I would research deployment platform restrictions before choosing one — knowing that PythonAnywhere free tier blocks external database connections upfront would have saved hours of troubleshooting. I would also test every template locally before deploying since catching the `split` filter error locally would have been much faster than debugging it through Railway logs. Overall converting from Streamlit to Django gave me a much deeper understanding of how real web applications are structured and deployed, and the live app at `https://web-production-f731e.up.railway.app` is something I can actually show to someone outside of class.
